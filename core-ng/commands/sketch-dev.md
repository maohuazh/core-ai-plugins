# 快速开发工作流

精简但完整的开发流程，适用于中小型功能。核心目标：快速输出高质量代码，不跳过关键步骤。

## 适用判断

**适用于**：
- 单模块或少模块变更
- 需求明确，实现路径清晰
- 不涉及复杂架构决策
- 时间敏感但需基本质量保障

**不适用于**：
- 跨多系统的复杂功能
- 需要深度代码库分析
- 需要多方案比较决策

---

## 阶段 1：需求澄清（5分钟）

**目标**：快速确认需求核心

**执行步骤**：
1. 解析 `$ARGUMENTS`
2. 简要确认：
   - 解决什么问题？
   - 核心功能点？
   - 有无特殊约束？
3. 如需求清晰，直接进入下一阶段

**输出**：简要需求记录

---

## 阶段 2：架构设计（10分钟）

**目标**：快速输出单一方案

**执行步骤**：
1. 启动 1 个 `code-architect` 代理：
   - 分析代码库现有模式
   - 设计最小变更方案
   - 输出实现蓝图
2. 阅读代理识别的关键文件
3. 向用户呈现：
   - 核心架构选择
   - 文件变更清单（创建/修改）
   - 实现步骤（3-5步）
4. **等待用户确认方案**

**代理提示词**：
```
设计 [功能名] 的最小变更实现方案，分析代码库模式，输出文件清单和构建序列
```

**输出**：确认的实现方案

---

## 阶段 3：代码实现（核心阶段）

**目标**：按方案实现功能

**前置条件**：用户已确认方案

**执行步骤**：
1. **等待用户明确批准开始编码**
2. 将方案拆解为 1-5 个任务（TodoWrite）
3. 阅读关键文件建立上下文
4. 逐步实现：
   - 严格遵循代码库命名和结构约定
   - 每完成一个任务更新 todo 状态
   - 编写清晰的代码和必要的注释
5. 完成所有任务后进入审查

**实现原则**：
- 复用现有组件优先
- 单一职责，避免过度抽象
- 错误处理完整
- 关键逻辑添加英文注释

---

## 阶段 4：质量审查（10分钟）

**目标**：确保代码基本质量

**执行步骤**：
1. 启动 1 个 `code-reviewer` 代理：
   - 审查维度：功能正确性、代码质量、项目约定、Checkstyle
   - 重点检查：逻辑错误、边界处理、命名规范
2. 整合审查发现
3. 向用户呈现：
   - 高置信度问题（≥80）
   - Checkstyle 失败项
4. **询问用户处理方式**：
   - 立即修复
   - 稍后修复
   - 按原样继续
5. 根据用户决定处理问题

**代理提示词**：
```
审查最近的变更，检查功能正确性、代码质量和项目约定，执行Checkstyle校验
```

---

## 阶段 5：完成

**目标**：总结工作

**执行步骤**：
1. 标记所有 todo 完成
2. 简要总结：
   - 实现了什么
   - 创建/修改的文件列表
   - 代码审查结论
   - 后续建议（如有）

---

## 时间估算

| 阶段 | 预估时间 |
|------|----------|
| 需求澄清 | 5分钟 |
| 架构设计 | 10分钟 |
| 代码实现 | 核心工作 |
| 质量审查 | 10分钟 |
| 完成 | 2分钟 |

**总计**：约30分钟（不含核心编码时间）

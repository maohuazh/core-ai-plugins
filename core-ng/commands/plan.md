# 功能规划工作流

你将引导用户完成系统化的功能规划流程。核心目标：输出结构化的PLAN.md实现蓝图，确保用户充分理解需求、代码库和架构决策后再进入实现。

## 核心原则

- **先理解再规划**：必须深入理解代码库后再提出架构方案
- **消除所有歧义**：设计前必须澄清所有边界情况和技术细节
- **用户决策驱动**：架构方案由用户选择，任务拆解需用户确认
- **输出结构化蓝图**：PLAN.md必须包含具体文件路径和实现步骤

---

## 阶段 1：需求发现

**目标**：明确要构建什么，识别约束条件

**执行步骤**：
1. 解析 `$ARGUMENTS`（如为空则询问用户）
2. 识别缺失信息：
   - 解决什么问题？
   - 核心功能是什么？
   - 技术约束有哪些？（框架、SDK版本、兼容性）
   - 非功能需求？（性能、安全、可观测性）
3. 总结需求并与用户确认

**输出**：需求描述文档（记录在PLAN.md开头）

---

## 阶段 2：代码库探索

**目标**：深度理解相关代码和架构模式

**执行步骤**：
1. 并行启动 2-3 个 `code-explorer` 代理：
   - **代理1**：查找类似功能实现，追踪完整路径
   - **代理2**：映射相关领域架构层次和依赖
   - **代理3**：识别技术栈和第三方SDK用法（如涉及）
2. 等待代理返回
3. **阅读所有代理识别的必读文件**（这是关键步骤）
4. 总结发现：现有模式、可复用组件、技术约束

**代理提示词示例**：
```
追踪 [类似功能名] 的完整实现路径，从入口点到数据存储，输出执行路径和关键文件
```
```
映射 [相关模块] 的架构层次，识别各层边界和跨层调用模式
```

**输出**：代码库分析摘要（记录在PLAN.md）

---

## 阶段 3：澄清问题

**目标**：消除所有歧义，获得完整需求描述

**必须澄清的维度**：

| 维度 | 澄清内容 |
|------|----------|
| 边界条件 | 空值处理、边界值、超限场景 |
| 错误处理 | 异常类型、错误码、重试策略、降级方案 |
| 数据模型 | 数据结构、校验规则、存储方式、迁移策略 |
| 集成点 | API契约、数据格式、兼容性要求 |
| 性能需求 | 响应时间、并发量、缓存策略 |
| 安全需求 | 认证授权、数据加密、审计日志 |
| 向后兼容 | 现有接口变更、数据迁移、版本策略 |

**执行步骤**：
1. 基于代码库分析和需求，识别未明确点
2. **以结构化列表呈现所有问题**
3. **等待用户回答所有问题后再继续**
4. 如用户说"你觉得怎样最好"，给出建议并获取明确确认

**输出**：澄清记录（记录在PLAN.md）

---

## 阶段 4：架构设计

**目标**：设计多种实现方案，由用户选择

**执行步骤**：
1. 并行启动 2-3 个 `code-architect` 代理：
   - **最小变更方案**：最大复用，最小新增代码
   - **清晰架构方案**：最佳可维护性，优雅抽象
   - **务实平衡方案**：权衡速度与质量
2. 等待代理返回实现蓝图
3. 整合方案，形成你的专业判断
4. 向用户呈现：
   - 每种方案的简要总结
   - 权衡比较（复杂度、风险、维护成本）
   - **你的推荐及理由**
   - 具体差异对比
5. **询问用户选择哪种方案**

**输出**：用户选择的架构方案（记录在PLAN.md）

---

## 阶段 5：任务拆解

**目标**：将方案拆解为可独立执行的实现任务

**拆解原则**：
- 每个任务可独立审查和测试
- 任务间依赖关系明确标注
- 每个任务包含具体文件和预期变更
- 任务数量控制在 1-20 个

**执行步骤**：
1. 按用户选择的方案拆解任务
2. 创建任务列表（使用 TodoWrite）
3. 创建 PLAN.md 文件，包含完整内容
4. **询问用户是否确认方案**
5. **等待用户确认或调整**

**PLAN.md 必须包含的内容**：

```markdown
# 功能规划：[功能名称]

## 需求描述
[问题定义、核心功能、约束条件]

## 代码库分析
[现有模式、可复用组件、技术约束]

## 澄清记录
[所有歧义的决策]

## 架构方案
[用户选择的方案详细描述]
- 决策理由
- 文件变更清单
- 组件设计
- 数据流
- 构建序列

## 实现任务
| 序号 | 任务 | 涉及文件 | 依赖 | 预期变更 |
|------|------|----------|------|----------|
| 1 | ... | ... | 无 | ... |
| 2 | ... | ... | 1 | ... |

## 测试策略
[测试范围、类型、关键场景]

## 关键文件清单
[实现前必须阅读的文件]

## 风险与注意事项
[已知风险、技术债务、兼容性注意]
```

**输出**：PLAN.md 文件

---

## 工作流结束

规划完成后，用户可选择：
- **进入实现**：使用 `/implement` 命令读取 PLAN.md
- **保存方案**：PLAN.md 已保存在项目根目录
- **调整方案**：修改 PLAN.md 后再进入实现

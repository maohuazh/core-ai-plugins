# 全面开发工作流

完整的开发流程，适用于复杂功能。核心目标：确保需求完整理解、架构科学决策、代码高质量交付。

## 适用判断

**适用于**：
- 跨多模块/系统的功能
- 需要深度代码库分析
- 涉及重要架构决策
- 对质量有严格要求
- 需求存在不确定性

**不适用于**：
- 简单单模块变更
- 需求明确且路径清晰
- 时间极度敏感

---

## 阶段 1：需求发现

**目标**：全面理解需求，识别所有约束

**执行步骤**：
1. 解析 `$ARGUMENTS`
2. 识别缺失信息：
   - 问题定义：解决什么业务问题？
   - 功能范围：核心功能、边界功能、排除功能
   - 技术约束：框架、SDK版本、兼容性、平台
   - 非功能需求：性能、安全、可观测性、可用性
   - 时间约束：deadline、里程碑
3. 总结需求并与用户确认

---

## 阶段 2：代码库探索（深度）

**目标**：全面理解代码库架构和相关实现

**执行步骤**：
1. 并行启动 2-3 个 `code-explorer` 代理：
   - **代理1**：类似功能实现追踪
   - **代理2**：相关模块架构映射
   - **代理3**：技术栈和集成点分析
2. 阅读所有代理识别的必读文件
3. 深度分析：
   - 架构模式：分层、模块划分、依赖方向
   - 数据流：关键数据的流转路径
   - 技术债务：现有问题和改进机会
   - 可复用点：现有组件、工具类、配置

**代理提示词示例**：
```
深度追踪 [类似功能] 的完整实现，输出执行路径、关键组件、依赖关系和必读文件
```

---

## 阶段 3：澄清问题（关键阶段）

**目标**：消除所有歧义，获得完整准确的需求

**必须澄清的维度**：

| 维度 | 具体问题示例 |
|------|--------------|
| 边界条件 | 空输入如何处理？数量上限？时间边界？ |
| 错误处理 | 失败重试策略？降级方案？错误码定义？ |
| 数据模型 | 数据校验规则？存储位置？迁移策略？ |
| 集成接口 | API契约？数据格式？版本兼容？ |
| 并发场景 | 并发量预期？锁策略？幂等性要求？ |
| 性能指标 | 响应时间要求？吞吐量？缓存策略？ |
| 安全需求 | 认证方式？授权粒度？数据加密？审计日志？ |
| 向后兼容 | 现有接口变更策略？数据迁移方案？版本过渡？ |

**执行步骤**：
1. 基于代码库分析，系统化列出所有未明确点
2. **以结构化方式呈现问题**
3. **等待用户回答所有问题**
4. 记录所有决策

**注意**：此阶段不能跳过，问题越详细，后续实现越准确。

---

## 阶段 4：架构设计（多方案）

**目标**：设计多种方案，提供充分决策依据

**执行步骤**：
1. 并行启动 2-3 个 `code-architect` 代理：
   - **最小变更**：最大复用，最小新增，风险最低
   - **清晰架构**：最佳可维护性，优雅抽象，长期最优
   - **务实平衡**：权衡实现成本与架构质量
2. 阅读代理输出的完整蓝图
3. 整合分析：
   - 各方案的复杂度对比
   - 各方案的风险评估
   - 各方案的维护成本
4. 向用户呈现：
   - 每种方案的核心差异
   - 权衡分析表
   - **你的专业推荐及理由**
5. **询问用户选择哪种方案**

---

## 阶段 5：任务拆解

**目标**：将方案拆解为可执行任务序列

**执行步骤**：
1. 按用户选择的方案拆解任务：
   - 每个任务可独立审查
   - 标注依赖关系
   - 包含具体文件和变更内容
2. 创建任务列表（TodoWrite）
3. 向用户呈现任务拆解
4. **等待用户确认方案和任务拆解**

---

## 阶段 6：实现

**目标**：按确认方案实现

**前置条件**：用户已明确批准

**执行步骤**：
1. **等待用户明确批准开始编码**
2. 阅读关键文件建立上下文
3. 按任务列表实现：
   - 严格遵循架构方案
   - 遵循代码库命名和结构约定
   - 每完成一个任务更新 todo
4. 保持代码质量：清晰、可测试、文档必要处

---

## 阶段 7：测试验证

**目标**：全面验证功能正确性

**执行步骤**：
1. 启动 1-2 个 `code-tester` 代理：
   - 设计完整测试策略
   - 编写单元测试、集成测试
   - 执行测试并分析结果
2. 整合测试结果：
   - 覆盖率分析
   - 失败测试诊断
   - 未覆盖场景识别
3. **向用户呈现测试结果**
4. 处理失败测试

---

## 阶段 8：质量审查（全面）

**目标**：多维度全面检查代码质量

**执行步骤**：
1. 并行启动 4 个 `code-reviewer` 代理：
   - **功能正确性**：逻辑错误、边界处理、并发安全、资源管理
   - **安全漏洞**：输入验证、认证授权、数据保护
   - **代码质量**：DRY原则、复杂度、可读性、测试覆盖
   - **项目约定**：CLAUDE.md规范、Checkstyle校验
2. 整合所有发现
3. 按置信度和严重性排序
4. **向用户呈现完整审查报告**：
   - 关键问题（立即修复）
   - 重要问题（建议修复）
   - Checkstyle失败
   - 良好实践建议
5. **询问用户处理方式**
6. 根据用户决定处理问题

---

## 阶段 9：总结

**目标**：完整记录交付成果

**执行步骤**：
1. 标记所有 todo 完成
2. 全面总结：
   - 构建内容清单
   - 关键决策回顾（架构选择、技术方案）
   - 修改的文件列表
   - 测试覆盖率统计
   - 代码审查结论
   - 风险与注意事项
   - 后续建议（扩展点、优化机会、文档更新）

---

## 时间估算

| 阶段 | 预估时间 |
|------|----------|
| 需求发现 | 10分钟 |
| 代码库探索 | 20分钟 |
| 澄清问题 | 15分钟 |
| 架构设计 | 15分钟 |
| 任务拆解 | 10分钟 |
| 实现 | 核心工作 |
| 测试验证 | 20分钟 |
| 质量审查 | 15分钟 |
| 总结 | 5分钟 |

**总计**：约1.5小时（不含核心编码时间）

## 与其他工作流的对比

| 特性 | sketch-dev | holistic-dev |
|------|------------|--------------|
| 探索深度 | 单代理快速分析 | 多代理深度追踪 |
| 澄清范围 | 核心问题 | 全维度系统化 |
| 架构方案 | 单一方案 | 多方案比较选择 |
| 测试验证 | 无 | 完整测试策略 |
| 质量审查 | 单代理 | 4代理全面审查 |
| 适用场景 | 中小型功能 | 复杂功能 |

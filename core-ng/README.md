# Core-ng 开发插件

专业化开发工作流插件，提供4个命令和4个代理，覆盖从需求分析到质量交付的全流程。

## 核心价值

- **流程适配复杂度**：简单任务精简流程，复杂功能完整流程
- **规划实现可分离**：支持独立规划、审批后再实现
- **代理专业分工**：代码追踪、架构设计、质量审查、测试验证各司其职
- **置信度驱动审查**：仅报告高置信度问题，消除噪音

---

## 命令一览

| 命令 | 阶段数 | 适用场景 | 核心特点 |
|------|--------|----------|----------|
| `/sketch-dev` | 4 | 中小型功能、需求明确 | 精简流程、快速交付 |
| `/holistic-dev` | 9 | 复杂功能、跨系统 | 深度分析、多方案设计 |
| `/plan` | 5 | 需单独审批的规划 | 输出PLAN.md、不含实现 |
| `/implement` | 6 | 已有方案的实现 | 基于PLAN.md、含测试审查 |

---

## 命令详解

### `/sketch-dev` - 快速开发

**适用判断**：
- ✅ 单模块或少模块变更
- ✅ 需求明确，实现路径清晰
- ✅ 不涉及复杂架构决策
- ❌ 跨多系统的复杂功能

**工作流程**：

| 阶段 | 目标 | 输出 |
|------|------|------|
| 1. 需求澄清 | 确认核心需求 | 需求记录 |
| 2. 架构设计 | 单一方案输出 | 实现方案 |
| 3. 代码实现 | 按方案编码 | 实现代码 |
| 4. 质量审查 | 多维度检查 | 审查报告 |

**代理配置**：
- 架构设计：1个 `code-architect`
- 质量审查：1个 `code-reviewer`（多维度整合）

**时间估算**：约30分钟（不含编码）

---

### `/holistic-dev` - 全面开发

**适用判断**：
- ✅ 跨多模块/系统的功能
- ✅ 需要深度代码库分析
- ✅ 涉及重要架构决策
- ✅ 对质量有严格要求
- ❌ 简单单模块变更

**工作流程**：

| 阶段 | 目标 | 输出 |
|------|------|------|
| 1. 需求发现 | 全面理解需求 | 需求文档 |
| 2. 代码库探索 | 深度分析代码 | 代码库分析 |
| 3. 澄清问题 | 消除所有歧义 | 澄清记录 |
| 4. 架构设计 | 多方案比较 | 用户选择的方案 |
| 5. 任务拆解 | 拆解实现任务 | 任务清单 |
| 6. 实现 | 按方案编码 | 实现代码 |
| 7. 测试验证 | 验证正确性 | 测试报告 |
| 8. 质量审查 | 4维度全面检查 | 审查报告 |
| 9. 总结 | 记录成果 | 交付总结 |

**代理配置**：
- 代码库探索：2-3个 `code-explorer`（并行）
- 架构设计：2-3个 `code-architect`（多方案）
- 测试验证：1-2个 `code-tester`
- 质量审查：4个 `code-reviewer`（并行多维度）

**时间估算**：约1.5小时（不含编码）

---

### `/plan` - 功能规划

**核心定位**：仅规划，不含实现。输出结构化PLAN.md供后续执行或审批。

**工作流程**：

| 阶段 | 目标 | 输出 |
|------|------|------|
| 1. 需求发现 | 明确需求和约束 | 需求描述 |
| 2. 代码库探索 | 深度理解现有代码 | 代码库分析 |
| 3. 澄清问题 | 系统化消除歧义 | 澄清记录 |
| 4. 架构设计 | 多方案比较选择 | 用户选择的方案 |
| 5. 任务拆解 | 输出实现蓝图 | PLAN.md |

**PLAN.md 标准结构**：
```markdown
# 功能规划：[功能名称]

## 需求描述
## 代码库分析
## 澄清记录
## 架构方案
## 实现任务（表格：序号/任务/文件/依赖/变更）
## 测试策略
## 关键文件清单
## 风险与注意事项
```

---

### `/implement` - 功能实现

**核心定位**：基于PLAN.md执行实现，含测试验证和质量审查。

**前置条件**：
- 存在PLAN.md（或用户指定路径）
- 方案已批准

**工作流程**：

| 阶段 | 目标 | 输出 |
|------|------|------|
| 1. 加载规划 | 读取确认方案 | 方案确认 |
| 2. 实现准备 | 建立代码上下文 | todo列表 |
| 3. 代码实现 | 严格按方案执行 | 实现代码 |
| 4. 测试验证 | 验证功能正确 | 测试报告 |
| 5. 质量审查 | 全面检查 | 审查报告 |
| 6. 总结 | 记录成果 | 交付总结 |

**代理配置**：
- 测试验证：1-2个 `code-tester`
- 质量审查：4个 `code-reviewer`

## 代理详解

### `code-explorer` - 代码库深度追踪

**核心任务**：追踪执行路径，映射架构层次，输出结构化分析报告。

**输出格式**：
- 入口点定位（`file:line`）
- 执行路径（逐层调用链）
- 关键发现（设计模式、技术债务、优化机会）
- 必读文件清单（5-10个，按重要性排序）

**调用示例**：
```
追踪认证模块的完整实现路径，从登录API到token存储
```

---

### `code-architect` - 架构蓝图设计

**核心任务**：分析代码库模式，输出包含文件清单、组件设计、数据流、构建序列的完整蓝图。

**设计原则**：
- 果断决策（选择一种方案，明确理由）
- 模式继承（严格遵循现有约定）
- 最小变更（优先复用现有组件）

**输出格式**：
- 决策理由
- 发现的模式（`file:line`引用）
- 文件变更清单（表格：新建/修改）
- 组件设计（职责、依赖、接口、测试要点）
- 数据流（从输入到输出的完整流转）
- 构建序列（按依赖顺序）

---

### `code-reviewer` - 置信度驱动审查

**核心任务**：多维度检测问题，基于置信度过滤，仅报告高优先级。

**审查维度**：
1. **功能正确性**：逻辑错误、空值处理、并发问题、资源管理
2. **安全漏洞**：输入验证、认证授权、数据保护
3. **代码质量**：DRY原则、复杂度、可读性、测试覆盖
4. **项目约定**：CLAUDE.md规范、命名、目录结构
5. **规范校验**：Checkstyle执行

**置信度评分**：

| 分数 | 含义 | 策略 |
|------|------|------|
| 0-79 | 低置信度 | **不报告**（过滤噪音） |
| 80-89 | 确定问题 | **必须报告** |
| 90-99 | 严重问题 | **必须报告**，优先修复 |
| 100 | 致命问题 | **必须报告**，立即修复 |

---

### `code-tester` - 测试策略与验证

**核心任务**：制定测试策略，设计测试用例，执行验证，输出覆盖率分析。

**测试类型**：
- **单元测试**：独立函数/方法
- **集成测试**：组件交互、API契约
- **边界测试**：空值、边界值、并发场景

**输出格式**：
- 测试策略（范围、类型、优先级）
- 测试用例清单（表格：用例名/场景/输入/预期/状态）
- 覆盖率分析（已覆盖/待覆盖）
- 风险提示（高风险未覆盖区域）

## 命令选择决策树

```
任务复杂度判断
├── 简单/中型？
│   ├── 需求明确？ → /sketch-dev
│   └── 需求模糊？ → 先澄清再选择
├── 复杂？
│   ├── 需单独审批？ → /plan + /implement
│   └── 流程内完成？ → /holistic-dev
└── 已有方案？ → /implement
```

## 代理手动调用

除命令自动触发外，可独立调用代理：

**探索代码**：
```
启动 code-explorer 追踪 [模块名] 的数据流转路径
```

**设计架构**：
```
启动 code-architect 设计 [功能名] 的最小变更方案
```

**审查代码**：
```
启动 code-reviewer 审查 src/auth/ 目录，执行Checkstyle
```

**验证测试**：
```
启动 code-tester 为 [功能名] 设计测试策略，重点覆盖边界场景
```

## 最佳实践

1. **匹配流程与复杂度**：简单任务用精简流程，避免过度分析
2. **澄清阶段必回答**：澄清质量决定后续实现准确度
3. **阅读必读文件**：代理输出的文件清单是理解上下文的关键
4. **方案选择需权衡**：多方案设计提供选择，慎重决策
5. **关注高置信度问题**：审查仅报告≥80置信度，这些必须处理
6. **测试验证不可跳**：测试确保功能正确和边界覆盖

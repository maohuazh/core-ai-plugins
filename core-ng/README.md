# Core-ng 开发插件

新一代开发工作流插件，提供规划、快速开发、实现三个命令，配备代码探索、架构设计、代码审查、测试验证四个专业代理。

## 概述

Core-ng 插件提供灵活的开发工作流选择，从精简的快速开发流程到完整的规划-实现分离模式，满足不同场景的开发需求。它通过专业代理系统实现代码库深度理解、架构设计、质量审查和测试验证，确保代码质量与项目规范的一致性。

## 理念

现代软件开发需要多样化的工作流支持：

- **不同复杂度的任务需要不同流程** - 简单任务用精简流程，复杂功能用完整流程
- **规划与实现可以分离** - 支持先规划后实现，也支持一步到位
- **专业分工提升效率** - 代码探索、架构设计、质量审查各有专攻
- **置信度驱动的质量审查** - 只报告真正重要的问题，减少噪音

此插件通过三个命令和四个代理，提供适配各种开发场景的工作流选择。

## 命令

### `/plan` - 功能规划

专注于规划阶段的开发流程，从需求发现到任务拆解，不包含实际实现。

**用法：**
```bash
/plan <功能描述>
```

或直接：
```bash
/plan
```

**适用场景：**
- 需要深思熟虑的复杂功能设计
- 需要与团队讨论方案后再实现
- 需要多个实现方案的比较选择
- 规划阶段需要单独审批的项目

**工作流程：**

| 阶段 | 目标 | 核心活动 |
|------|------|----------|
| 1. 发现 | 了解需求 | 澄清功能、识别约束、确认理解 |
| 2. 代码库探索 | 理解现状 | 启动 2-3 个 code-explorer 代理并行探索 |
| 3. 澄清问题 | 消除歧义 | 提出边界情况、错误处理、集成点等问题 |
| 4. 架构设计 | 方案选择 | 启动 2-3 个 code-architect 代理设计不同方案 |
| 5. 任务拆解 | 输出蓝图 | 创建 PLAN.md 文件，拆分为 1-20 个可独立任务 |

**输出交付物：**
- 需求文档：清晰的问题定义和功能描述
- 代码库分析：相关现有代码、模式、依赖的深入理解
- 澄清记录：所有歧义和边缘情况的决策
- 架构方案：详细的实现蓝图，包含文件路径、组件设计、数据流
- 任务清单：可独立执行的实现任务列表，带依赖关系

### `/sketch-dev` - 快速开发

精简的 4 阶段开发流程，配备设计和测试代理，快速完成中小型功能开发。

**用法：**
```bash
/sketch-dev <功能描述>
```

或直接：
```bash
/sketch-dev
```

**适用场景：**
- 中小型功能开发
- 需求明确、实现路径清晰的任务
- 时间紧迫但需保证基本质量
- 不需要复杂架构决策的功能

**工作流程：**

| 阶段 | 目标 | 核心活动 |
|------|------|----------|
| 1. 需求澄清 | 明确目标 | 简要确认需求、识别核心功能 |
| 2. 架构设计 | 快速方案 | 启动 1 个 code-architect 代理输出方案 |
| 3. 代码实现 | 按方案编码 | 拆分 1-5 个任务、逐步实现 |
| 4. 质量审查 | 基本检查 | 启动 1 个 code-reviewer 代理多维度审查 |

**特点：**
- 精简但不跳过关键步骤
- 单代理设计，快速输出
- 多维度代码审查集中在一个代理
- 适合快速迭代场景

### `/implement` - 功能实现

基于已有规划文档直接进入实现阶段，跳过分析和设计。

**用法：**
```bash
/implement [规划文档路径]
```

默认读取项目根目录的 `PLAN.md`。

**前置条件：**
- 已完成规划阶段（使用 `/plan` 命令）
- 存在规划文档（默认为 `PLAN.md`）

**适用场景：**
- 规划已完成，需要单独执行实现
- 规划和实现由不同人员负责
- 需要按已有方案快速推进
- 方案已经过评审和批准

**工作流程：**

| 阶段 | 目标 | 核心活动 |
|------|------|----------|
| 1. 加载规划 | 读取方案 | 确认 PLAN.md 内容、询问用户批准 |
| 2. 实现准备 | 建立上下文 | 阅读关键文件、创建 todo 列表 |
| 3. 代码实现 | 按方案编码 | 逐步实现任务、遵循代码库约定 |
| 4. 测试验证 | 验证功能 | 启动 1-2 个 code-tester 代理 |
| 5. 质量审查 | 代码检查 | 启动 4 个 code-reviewer 代理并行审查 |
| 6. 总结 | 记录完成 | 标记完成、总结工作、清理规划文档 |

---

### `/holistic-dev` - 全面开发

完整的 9 阶段开发流程，包含从需求发现到测试验证的全流程。

**用法：**
```bash
/holistic-dev <功能描述>
```

或直接：
```bash
/holistic-dev
```

**适用场景：**
- 复杂功能开发，需要全面考量
- 涉及多个模块的跨系统功能
- 需要深度代码库分析的任务
- 对质量有严格要求的功能

**工作流程：**

| 阶段 | 目标 | 核心活动 |
|------|------|----------|
| 1. 发现 | 了解需求 | 澄清功能、识别约束 |
| 2. 代码库探索 | 深度理解 | 启动 2-3 个 code-explorer 代理 |
| 3. 澄清问题 | 消除歧义 | 提出所有未明确细节 |
| 4. 架构设计 | 方案选择 | 启动 2-3 个 code-architect 代理 |
| 5. 任务拆解 | 输出蓝图 | 拆分任务、等待用户确认 |
| 6. 实现 | 构建功能 | 按方案实现、遵循约定 |
| 7. 测试验证 | 验证正确 | 启动 1-2 个 code-tester 代理 |
| 8. 质量审查 | 全面检查 | 启动 4 个 code-reviewer 代理 |
| 9. 总结 | 记录完成 | 总结工作、标记完成 |

## 代理

### `code-explorer` - 代码探索

**用途：** 通过追踪执行路径深入分析现有代码库功能，为新开发提供信息。

**模型：** glm-5
**颜色标识：** yellow

**核心能力：**
- 功能发现：查找入口点（API、UI 组件、CLI 命令）
- 代码流追踪：从入口到输出跟踪调用链
- 架构分析：映射抽象层、识别设计模式
- 依赖记录：识别所有内部和外部依赖

**输出内容：**
- 带 file:line 引用的入口点
- 带数据转换的逐步执行流程
- 关键组件及其职责
- 架构洞察：模式、层、设计决策
- 必读文件列表（5-10个关键文件）

---

### `code-architect` - 架构设计

**用途：** 通过分析现有代码库模式和约定设计功能架构，提供全面的实现蓝图。

**模型：** sonnet
**颜色标识：** green

**核心能力：**
- 代码库模式分析：提取现有模式、约定和架构决策
- 架构设计：做出果断选择，确保与现有代码无缝集成
- 完整实现蓝图：指定每个文件、组件职责、数据流
- 构建序列：分阶段实现步骤作为清单

**输出内容：**
- 发现的模式和约定（带 file:line 引用）
- 架构决策及理由和权衡
- 组件设计：文件路径、职责、依赖和接口
- 实现映射：要创建/修改的具体文件
- 数据流：从入口点经过转换到输出
- 构建序列：分阶段实现步骤

---

### `code-reviewer` - 代码审查

**用途：** 审查代码的 bug、逻辑错误、安全漏洞、代码质量问题和项目约定遵循情况。

**模型：** sonnet
**颜色标识：** red

**核心能力：**
- 项目指南合规性：验证对 CLAUDE.md 中项目规则的遵循
- Bug 检测：识别逻辑错误、null/undefined 处理、竞态条件、内存泄漏、安全漏洞
- 代码质量：评估代码重复、错误处理、测试覆盖
- Checkstyle 校验：执行代码规范检查

**置信度评分系统：**

| 分数 | 含义 | 报告策略 |
|------|------|----------|
| 0 | 完全无信心，误报 | 不报告 |
| 25 | 有些信心，可能是误报 | 不报告 |
| 50 | 中等信心，可能不重要 | 不报告 |
| 75 | 高信心，真实问题 | 不报告 |
| 80+ | 确定是重要问题 | **必须报告** |
| 100 | 绝对确定，频繁发生的问题 | **必须报告** |

**输出内容：**
- 仅报告置信度 ≥ 80 的问题
- Checkstyle 失败必须显示
- 按严重性分组（关键 vs 重要）
- 带文件路径和行号的具体修复建议

---

### `code-tester` - 测试验证

**用途：** 编写和执行测试用例，验证功能正确性、边界条件和集成稳定性。

**模型：** glm-5
**颜色标识：** cyan

**核心能力：**
- 测试策略制定：分析需求、确定测试范围和优先级
- 测试用例设计：功能测试、边界测试、异常测试、集成测试
- 测试执行与验证：运行测试、分析失败、验证修复

**测试类型支持：**
- **单元测试**：测试单个函数/方法，使用项目测试框架
- **集成测试**：测试组件交互，验证 API 契约和数据流
- **E2E 测试**：测试完整用户场景，验证系统端到端行为

**输出内容：**
- 测试策略：测试范围、类型选择、优先级排序
- 测试用例清单：描述、输入、预期输出
- 测试覆盖率分析：已覆盖场景、待覆盖场景
- 测试执行结果：通过/失败状态、失败原因分析
- 风险提示：高风险区域、建议额外测试的场景

## 命令选择指南

根据任务特征选择合适的命令：

| 任务特征 | 推荐命令 | 理由 |
|----------|----------|------|
| 简单功能、需求明确 | `/sketch-dev` | 精简流程，快速完成 |
| 中等复杂度、需基本设计 | `/sketch-dev` | 4阶段足够保证质量 |
| 复杂功能、需深度分析 | `/holistic-dev` | 完整流程确保全面考量 |
| 跨多模块、架构决策重要 | `/holistic-dev` | 多代理设计提供方案比较 |
| 规划需单独评审 | `/plan` + `/implement` | 规划实现分离，支持审批流程 |
| 方案已批准、直接实现 | `/implement` | 跳过规划阶段，提高效率 |

## 最佳实践

1. **选择匹配的命令** - 不要对简单任务用完整流程，也不要对复杂任务用精简流程
2. **认真回答澄清问题** - 澄清阶段的问题直接影响后续设计质量
3. **阅读代理识别的文件** - 代理输出的必读文件列表是理解代码库的关键
4. **慎重选择架构方案** - 架构设计阶段提供多方案是有原因的，权衡后再选择
5. **关注高置信度问题** - code-reviewer 仅报告置信度 ≥80 的问题，这些值得修复
6. **不跳过测试验证** - code-tester 确保功能正确性和边界覆盖

| 特性 | `/sketch-dev` | `/holistic-dev` | `/plan`+`/implement` |
|------|---------------|-----------------|---------------------|

| 测试代理 | 0 | 1-2 | 1-2 |

## 故障排除

### 代理运行时间过长

**原因**：大型代码库需要更多时间探索

**解决方案**：
- 这是正常现象，彻底性以更好的理解为回报
- 代理尽可能并行运行以减少等待
- 可在功能描述中指定范围缩小探索区域

### 澄清问题太多

**原因**：功能描述不够具体

**解决方案**：
- 在初始请求中提供更多细节
- 提前说明约束和技术偏好
- 如真的无偏好，可说"你觉得怎样最好"

### 测试代理报告覆盖率不足

**原因**：新功能测试用例不完整

**解决方案**：
- 查看测试代理输出的"待覆盖场景"
- 关注"高风险区域"提示
- 按建议补充测试用例

# Holistic Development 插件

一个全面、结构化的功能开发流程，配备专门的代理用于需求分析、Prompt优化、代码库探索、架构设计和质量审查。

## 概述

Holistic Development 插件提供系统的 8 阶段方法来构建新功能。它直接跳入代码，而是引导您优化需求描述、了解代码库、提出澄清问题、设计架构并确保质量--从而产生与现有代码无缝集成的最优结果。

## 理念

构建功能需要的不仅仅是编写代码。您需要:
- **优化需求描述**
- **更改前分析代码库**
- **澄清问题**以获得更准确、更完整的需求描述
- **边界问题的探索**
- **深思熟虑的架构设计**
- **永不缺席的质量审查**

此插件将这些实践嵌入结构化工作流程，在使用 `/holistic-dev` 命令时自动运行。

## 命令：`/holistic-dev`

启动带有 7 阶段的引导式功能开发工作流程。

**用法：**
```bash
/holistic-dev <功能描述>
```

或直接：
```bash
/holistic-dev
 ```

命令将以交互式方式引导您完成整个过程。

## 8 阶段工作流程

### 阶段 1：发现

**目标**：了解需要构建什么

**发生什么：**
- 如果功能请求不清楚则澄清
- 询问您正在解决什么问题
- 识别约束和要求
- 总结理解并与您确认

**示例：**
```
您：/holistic-dev 添加缓存
LLM：让我了解您需要什么...
        - 应该缓存什么？（API 响应、计算值等）
        - 您的性能要求是什么？
        - 您有首选的缓存解决方案吗？
```

### 阶段 2：代码库探索

**目标**：分析相关现有代码和模式

**发生什么：**
- 并行启动 2-3 个 `code-explorer` 代理
- 每个代理探索不同方面（类似功能、架构、UI 模式）
- 代理返回全面分析及要阅读的关键文件
- LLM 阅读所有识别的文件以建立深入理解
- 呈现全面的发现摘要

**启动的代理：**
- "查找与 [功能] 类似的功能并追踪实现"
- "映射 [领域] 的架构和抽象"
- "分析 [相关功能] 的当前实现"
- "识别 [相关第三方SDK] 的数据模型和方法"

**示例输出：**
```
发现的类似功能：
- 用户认证 (src/auth/)：使用 JWT token、中间件模式
- 会话管理 (src/session/)：Redis 支持、24小时过期
- API 安全 (src/api/middleware/)：速率限制、CORS

要理解的关键文件：
- src/auth/AuthService.ts:45 - 核心认证逻辑
- src/middleware/authMiddleware.ts:12 - 请求认证
- src/config/security.ts:8 - 安全配置
```

### 阶段 3：澄清问题

**目标**：填补空白并解决所有歧义

**发生什么：**
- 审查代码库发现和功能请求
- 识别未明确的方面：
    - 边缘情况
    - 错误处理
    - 集成点
    - 向后兼容性
    - 性能需求
    - 第三方SDK版本
- 以有组织的列表呈现所有问题
- **在继续前等待您的回答**

**示例：**
```
在设计架构之前，我需要澄清：

1. OAuth 提供商：哪些 OAuth 提供商？（Google、GitHub、自定义？）
2. 用户数据：存储 OAuth token 还是仅存储用户配置文件？
3. 现有认证：替换当前认证还是并行添加？
4. 会话：与现有会话管理集成？
5. 错误处理：如何处理 OAuth 失败？
```

**关键**：此阶段确保设计开始前没有任何歧义。

### 阶段 4：架构设计

**目标**：设计多种实现方法

**发生什么：**
- 启动 2-3 个不同重点的 `code-architect` 代理：
    - **最小更改**：最小改动、最大重用
    - **清晰架构**：可维护性、优雅抽象
    - **务实平衡**：速度 + 质量
- 审查所有方法
- 形成最适合此特定任务的意见
- 呈现比较及权衡和建议
- **询问您偏好哪种方法**

**示例输出：**
```
我设计了 3 种方法：

方法 1：最小更改
- 用 OAuth 方法扩展现有 AuthService
- 向现有 auth router 添加新的 OAuth 路由
- 需要最少重构
优点：快速、低风险
缺点：OAuth 与现有认证耦合、更难测试

方法 2：清晰架构
- 新 OAuthService 带专用接口
- 独立的 OAuth router 和中间件
- 重构 AuthService 使用通用接口
优点：清晰分离、可测试、可维护
缺点：更多文件、更多重构

方法 3：务实平衡
- 新 OAuthProvider 抽象
- 集成到现有 AuthService
- 最少重构、良好边界
优点：平衡复杂性和清晰度
缺点：仍有一些耦合

建议：方法 3 - 给您清晰边界而无需
过多重构，且很适合您现有架构。

您想使用哪种方法？
```

### 阶段 5：任务拆解

**目标**：将用户确认的方案拆分为 1~20 个可独立的任务

**发生什么：**
- **将方案拆分为任务**
- 每个任务严格遵循可独立审查
- 编写清晰的任务文档
- **完成后询问是否要保存方案**

### 阶段 6：实现

**目标**：构建功能

**发生什么：**
- **开始前等待明确批准**
- 阅读前几个阶段识别的所有相关文件
- 按选定架构实现
- 严格遵循代码库约定
- 编写清晰、文档良好的代码
- 进度更新 todos

**注意：**
- 实现仅在您批准后开始
- 遵循阶段 2 发现的模式
- 使用阶段 4 设计的架构
- 持续跟踪进度

### 阶段 7：质量审查

**目标**：确保代码简单、DRY、优雅且功能正确

**发生什么：**
- 并行启动 3 个不同重点的 `code-reviewer` 代理：
    - **简洁性/DRY/优雅性**：代码质量和可维护性
    - **Bug/正确性**：功能正确性和逻辑错误
    - **约定/抽象**：项目标准和模式
    - **Checkstyle**：代码规范校验
- 整合发现
- 识别最高严重性问题
- 识别Checkstyle的所有失败
- **呈现发现并询问您想做什么**：
    - 现在修复
    - 稍后修复
    - 按原样继续
- 根据您的决定处理问题

**示例输出：**
```
代码审查结果：

高优先级问题：
1. OAuth 回调中缺少错误处理 (src/auth/oauth.ts:67)
2. 内存泄漏：OAuth state 未清理 (src/auth/oauth.ts:89)
3. 按照次优的方式排序 (src/**/service.java:153)

中等优先级：
1. 可简化 token 刷新逻辑 (src/auth/oauth.ts:120)
2. 考虑提取 OAuth 配置验证

所有测试通过。代码遵循项目约定。

您想怎么做？
```

### 阶段 8：总结

**目标**：记录完成的工作

**发生什么：**
- 标记所有 todos 完成
- 标记所有 任务 完成
- 总结：
    - 构建了什么
    - 做出的关键决策
    - 修改的文件
    - 建议的后续步骤

**示例：**
```
功能完成：OAuth 认证

构建内容：
- 支持 Google 和 GitHub 的 OAuth 提供商抽象
- 与现有认证集成的 OAuth 路由和中间件
- Token 刷新和会话集成
- 所有 OAuth 流程的错误处理

关键决策：
- 使用 OAuthProvider 抽象的务实方法
- 与现有会话管理集成
- 添加 OAuth state 防止 CSRF

修改的文件：
- src/auth/OAuthProvider.ts（新建）
- src/auth/AuthService.ts
- src/routes/auth.ts
- src/middleware/authMiddleware.ts

建议后续步骤：
- 为 OAuth 流程添加测试
- 添加更多 OAuth 提供商（Microsoft、Apple）
- 更新文档
```

## 代理

### `code-explorer`

**用途**：通过追踪执行路径深入分析现有代码库功能

**重点领域：**
- 入口点和调用链
- 数据流和转换
- 架构层和模式
- 依赖和集成
- 实现细节

**触发时机：**
- 阶段 2 自动
- 探索代码时可手动调用

**输出：**
- 带 file:line 引用的入口点
- 逐步执行流程
- 关键组件和职责
- 架构洞察
- 必读文件列表

### `code-architect`

**用途**：设计功能架构和实现蓝图

**重点领域：**
- 代码库模式分析
- 架构决策
- 组件设计
- 实现路线图
- 数据流和构建序列

**触发时机：**
- 阶段 4 自动
- 架构设计时可手动调用

**输出：**
- 发现的模式和约定
- 带理由的架构决策
- 完整组件设计
- 带具体文件的实现映射
- 分阶段的构建序列

### `code-reviewer`

**用途**：审查代码的 bug、质量问题和项目约定

**重点领域：**
- 项目指南合规性（CLAUDE.md）
- Bug 检测
- 代码质量问题
- 基于置信度的过滤（仅报告高置信度问题 ≥80）
- 执行Checkstyle校验

**触发时机：**
- 阶段 7 自动
- 编写代码后可手动调用

**输出：**
- 关键问题（置信度 75-100）
- 重要问题（置信度 50-74）
- 带 file:line 引用的具体修复
- 项目指南引用

## 使用模式

### 完整工作流（新功能推荐）：
```bash
/feature-dev 为 API 端点添加速率限制
```

让工作流引导您完成所有 7 个阶段。

### 手动代理调用：

**探索功能：**
```
"启动 code-explorer 追踪认证如何工作"
```

**设计架构：**
```
"启动 code-architect 设计缓存层"
```

**审查代码：**
```
"启动 code-reviewer 检查我最近的更改"
```

## 最佳实践

1. **对复杂功能使用完整工作流**：8 个阶段确保全面规划
2. **认真回答澄清问题**：阶段 3 防止未来困惑
3. **慎重选择架构**：阶段 4 给您选项是有原因的
4. **不要跳过代码审查**：阶段 7 在问题进入生产前捕获
5. **阅读建议的文件**：阶段 2 识别关键文件 — 阅读它们以理解上下文

## 何时使用此插件

**用于：**
- 涉及多个文件的新功能
- 需要架构决策的功能
- 与现有代码的复杂集成
- 需求有些不明确的功能

**不用于：**
- 单行 bug 修复
- 琐碎更改
- 明确定义的简单任务
- 紧急热修复

## 要求

- 已安装 Claude Code
- Git 仓库（用于代码审查）
- 有现有代码库的项目（工作流假设有现有代码可学习）

## 故障排除

### 代理花费时间太长

**问题**：代码探索或架构代理很慢

**解决方案**：
- 对于大型代码库这是正常的
- 代理尽可能并行运行
- 彻底性以更好的理解为回报

### 澄清问题太多

**问题**：阶段 3 问太多问题

**解决方案**：
- 在初始功能请求中更具体
- 提前提供约束上下文
- 如真的无偏好说"你觉得怎样最好"

### 架构选项令人不知所措

**问题**：阶段 4 架构选项太多

**解决方案**：
- 信任建议 — 基于代码库分析
- 如仍不确定，要求更多解释
- 如有疑问选择务实选项

## 提示

- **在功能请求中具体**：更多细节 = 更少澄清问题
- **信任流程**：每个阶段建立在前一个阶段上
- **审查代理输出**：代理提供关于代码库的宝贵洞察
- **不要跳过阶段**：每个阶段都有目的
- **用于学习**：探索阶段教您关于自己代码库的知识
